  1: <?php namespace ProcessWire;

  3: /**
  4:  * ProcessWire Module Interface
  5:  *
  6:  * Provides the base interfaces required by modules.
  7:  * 
  8:  * ProcessWire 3.x, Copyright 2018 by Ryan Cramer
  9:  * https://processwire.com
 10:  * 
 11:  * #pw-summary Module is the primary PHP interface for module types in ProcessWire. 
 12:  * #pw-body = 
 13:  * The Module interface doesn't actually require any specific methods,
 14:  * (other than the `className()` method) but is required as an interface 
 15:  * to state your intention to ProcessWire that your class is to be used 
 16:  * as a Module. As a result, all methods are optional, but including the 
 17:  * Module interface is not. You must also provide a means by which ProcessWire
 18:  * can query information about your module. More on that below. 
 19:  * 
 20:  * ### Implementing the Module interface
 21:  * 
 22:  * Below is how you indicate a PHP class is a ProcessWire Module:
 23:  * 
 24:  * ~~~~~
 25:  * <?php namespace ProcessWire;
 26:  * class HelloWorld extends WireData implements Module { 
 27:  *   // your class implementation
 28:  * }
 29:  * ~~~~~
 30:  * 
 31:  * Modules should either extend the `WireData` class, or if they are a 
 32:  * predefined type already recognized by ProcessWire, they should extend 
 33:  * the base class of that type (or another module based upon it). Base 
 34:  * module types include:
 35:  *
 36:  * - `AdminTheme`
 37:  * - `Fieldtype`
 38:  * - `FileCompilerModule`
 39:  * - `FileValidatorModule`
 40:  * - `Inputfield`
 41:  * - `ModuleJS`
 42:  * - `PageAction`
 43:  * - `Process`
 44:  * - `Textformatter`
 45:  * - `WireAction`
 46:  * - `WireMail`
 47:  * - `WireSessionHandler`
 48:  * 
 49:  * ### Requirements for Modules
 50:  * 
 51:  * 1. Must provide a means of getting information about the module.
 52:  *    This can be with a static `getModuleInfo()` method, or with
 53:  *    a `ModuleName.info.php` file or a `ModuleName.info.json` file. 
 54:  * 
 55:  * 2. Must provide a `className()` method that returns the module 
 56:  *    class name. All Wire derived objects already do this, so 
 57:  *    you don't have to provide it unless your Module does not
 58:  *    descend from a ProcessWire class. We recommend that your
 59:  *    modules extend the Wire or WireData class. 
 60:  * 
 61:  * 3. If you have a `__construct()` method, it must not require any
 62:  *    particular arguments. 
 63:  * 
 64:  * 4. If your module is configurable, it must also fulfill the
 65:  *    `ConfigurableModule` interface. 
 66:  * 
 67:  * ### Optional methods 
 68:  * 
 69:  * - `__construct()` - Called before module config is populated.
 70:  * 
 71:  * - `init()` - Called after module config is populated.
 72:  * 
 73:  * - `ready()` - Called after init(), after API ready. Note that ready() applies to 'autoload' modules only. 
 74:  * 
 75:  * - `install()` - Called when module is installed. 
 76:  * 
 77:  * - `uninstall()` - Called when module is uninstalled. 
 78:  * 
 79:  * - `upgrade($fromVersion, $toVersion)` - Called on version change.
 80:  * 
 81:  * - `isAutoload()` - Returns a boolean indicating whether the module should be loaded at boot. 
 82:  *   Can also be specified as a property in module information.
 83:  * 
 84:  * - `isSingular()` - Returns a boolean indicating whether the module is limited to one instance or not. 
 85:  *   Can also be specified as a property in module information.
 86:  * 
 87:  * - `getModuleInfo()` - A static method that returns an array of module information. 
 88:  * 
 89:  * These methods are outlined in more detail further down on this page. 
 90:  * 
 91:  * -----------------------------------------------------------------
 92:  * 
 93:  * ## Module Information
 94:  * 
 95:  * Modules must have some way to communicate information about 
 96:  * themselves to ProcessWire. This is done by providing an 
 97:  * associative array containing this module information. One
 98:  * of the following implementations is required: 
 99:  * 
100:  * 1. `getModuleInfo()` static method in your module class that returns an array.
101:  * 2. `YourModuleClass.info.php` file that populates an `$info` array.
102:  * 3. `YourModuleClass.info.json` file that contains an `info` object.
103:  *   
104:  * Each of these are demonstrated below:
105:  *  
106:  * **1) Static getModuleInfo() method:**
107:  * ~~~~~
108:  * public static function getModuleInfo() {
109:  *   return array(
110:  *     'title' => 'Your Module Title',
111:  *     'version' => 1,
112:  *     'author' => 'Your Name',
113:  *     'summary' => 'Description of what this module does and who made it.',
114:  *     'href' => 'http://www.domain.com/info/about/this/module/',
115:  *     'autoload' => false, // set to true if module should auto-load at boot
116:  *     'requires' => array(
117:  *        'HelloWorld>=1.0.1', 
118:  *        'PHP>=5.4.1', 
119:  *        'ProcessWire>=2.4.1'
120:  *     ),
121:  *     'installs' => array('Module1', 'Module2', 'Module3'),
122:  *     );
123:  * }
124:  * ~~~~~
125:  *
126:  * **2) YourModuleClass.info.php file:**  
127:  * Your file should populate an `$info` variable with an array exactly like 
128:  * described for #1 above, i.e.
129:  * 
130:  * ~~~~~
131:  * $info = array(
132:  *   'title' => 'Your Module Title',
133:  *   'version' => 1,
134:  *   // and so on, like the static version above
135:  * );
136:  * ~~~~~
137:  *
138:  * **3) YourModuleClass.info.json file:**  
139:  * Your JSON file should contain nothing but an object/map of the module info:
140:  * 
141:  * ~~~~~
142:  * {
143:  *   title: 'Your Module title',
144:  *   version: 1,
145:  *   // and so on
146:  * }
147:  * ~~~~~
148:  * 
149:  * -----------------------------------------------------------------
150:  *
151:  * ## Module information properties
152:  * 
153:  * ### Required info properties
154:  *
155:  * - `title` (string): The module's title.
156:  * 
157:  * - `version` (int|string): An integer or string that indicates the version number.
158:  * 
159:  * - `summary` (string): Summary text of the module (1 sentence recommended).
160:  * 
161:  * ### Optional info properties
162:  * 
163:  * - `href` (string): URL to more information about the module.
164:  * 
165:  * - `requires` (array|string): Array or CSV string of module class names that are required by this 
166:  *    module in order to install.
167:  * 
168:  *    - **Requires Module version:** If a particular version of the module is required, then specify an operator
169:  *      and version number after the module name, like this: "HelloWorld>=1.0.1". 
170:  * 
171:  *    - **Requires PHP version:** If a particular version of PHP is required, then specify "PHP" as the module name
172:  *      followed by an operator and required version number, like this: "PHP>=5.6.0". 
173:  * 
174:  *    - **Requires ProcessWire version:** If a particular version of ProcessWire is required, then specify
175:  *      ProcessWire followed by an operator and required version number, like this: "ProcessWire>=2.4.1". 
176:  * 
177:  * - `installs` (array|string): Array or CSV string of module class names that this module will handle install 
178:  *    and uninstall for.
179:  *    This causes ProcessWire's dependency checker to ignore them and it is assumed your module will handle 
180:  *    them. If your module does not handle them, ProcessWire will automatically install/uninstall them 
181:  *    immediately after your module.
182:  * 
183:  * - `permanent` (boolean): This property is intended for only for core modules. When true, a module cannot be uninstalled.
184:  * 
185:  * - `permission` (string): Name of permission required of a user before ProcessWire will load the module (for non-superusers).
186:  *    Note that ProcessWire will not install this permission if it doesn't yet exist. To have it installed automatically,
187:  *    see the _permissions_ option below this.
188:  * 
189:  * - `permissions` (array): Array of permissions that ProcessWire will install (and uninstall) automatically.
190:  *    Permissions should be in the format: array('permission-name' => 'Permission description'). 
191:  * 
192:  * - `icon` (string): Optional icon name to represent this module.
193:  *    Currently uses [font-awesome](http://fortawesome.github.io/Font-Awesome/) icon names.
194:  *    Omit the "fa-" part, leaving just the icon name.
195:  * 
196:  * - `singular` (boolean): Is only one instance of this module allowed? (default=auto-detect).
197:  *    This is good for any module that you want to eliminate the possibility of multiple instances
198:  *    running at once. For instance, modules that become API variables are typically singular, whereas
199:  *    something like an Inputfield module would not be singular. When not specified, modules that extend an 
200:  *    existing base type typically inherit the singular setting from the module they extend. 
201:  * 
202:  * - `autoload` (boolean|string|callable|int): Should this module load automatically at boot? (default=false).
203:  *    This is good for modules that attach hooks or that need to otherwise load on every single
204:  *    request. Autoload is typically specified as a boolean true or false. Below are the different ways
205:  *    autoload can be specified: 
206:  * 
207:  *    - **Boolean:** Specify true or false to indicate the module should either always autoload (true) 
208:  *      or never autoload (false). 
209:  * 
210:  *    - **Selector string:** The module will be automatically loaded only if the current page matches the
211:  *      selector string. For example, a selector string of `template=admin` would mean the module will
212:  *      only autoload in the admin side of ProcessWire. 
213:  * 
214:  *    - **Callable function:** The module will automatically load only if the given callable function 
215:  *      returns true. 
216:  * 
217:  *    - **Integer:** If given integer 2 or higher, it will autoload the module before other autoload
218:  *      modules (in /site/modules/). Higher numbers autoload before lower numbers. 
219:  * 
220:  * - `searchable` (string): When present, indicates that module implements a search() method 
221:  *    consistent with the SearchableModule interface. The value of the 'searchable' property should 
222:  *    be the name that the search results are referred to, using ascii characters of a-z, 0-9, and
223:  *    underscore. See the SearchableModule interface in this file for more details. 
224:  * 
225:  * -----------------------------------------------------------------------------------------------
226:  * 
227:  * ## Module Methods
228:  * 
229:  * ### __construct()
230:  * 
231:  * This method is called by PHP immediately when the module is instantiated, and before any 
232:  * configuration data has been populated to the module. This method must not have any required
233:  * arguments. This method is a good place for populating default configuration values or any
234:  * other initialization you want to occur before ProcessWire sees it or populates anything to it.
235:  * Your construct method should not assume that the module will actually be executed, as 
236:  * ProcessWire may instantiate a module for informational reasons. 
237:  * 
238:  * ### init()
239:  * 
240:  * This method is called after `__construct()` and after any configuration data has been populated
241:  * to the module. It is called before the module is handed over to the requester. This is a good
242:  * place to perform any initialization that requires configuration data and can be a good place to 
243:  * attach hooks. 
244:  * 
245:  * ### ready()
246:  * 
247:  * This method is used only by _autoload_ modules. It is called when the entire ProcessWire API
248:  * is ready to use. This may be preferable to the `init()` method for autoload modules because 
249:  * they are loaded and init()'d at boot, when everything else is loading too. The ready() method
250:  * is called once the boot has completed and all API variables are ready to use, but before any
251:  * page has been rendered. This makes it an excellent place to attach hooks. 
252:  * 
253:  * ### isSingular()
254:  *
255:  * Indicates whether only one instance of a module is allowed to exist in memory. 
256:  * If this method is not present, it will be auto-determined based on module type. If it is provided
257:  * in the module information array (discussed above) that will override this method. 
258:  * 
259:  * This method exists primarily so that base module types may specify a singular state and have it 
260:  * automatically inherit to any modules extending the type. If you are not extending a base module
261:  * type then you can implement this method, or you can provide it in your module info array. 
262:  *
263:  * A module that returns TRUE is referred to as a "singular" module, because there will never be any more
264:  * than a single instance of the module running.
265:  *
266:  * Return TRUE if this module is a single reusable instance, returning the same instance on every 
267:  * call from Modules. Return FALSE if this module should return a new instance on every call from Modules.
268:  *
269:  * - Singular modules will have their instance active for the entire request after instantiated.
270:  * - Non-singular modules return a new instance on every `$modules->get("YourModule")` call.
271:  * - Modules that attach hooks are usually singular.
272:  * - Modules that may have multiple instances (like `Inputfield` modules) should _not_be singular.
273:  *
274:  * If you are having trouble deciding whether to make your module singular or not, be sure to read 
275:  * the documentation below for the `isAutoload()` method, because if your module is 'autoload' then 
276:  * it's probably also 'singular'.
277:  * 
278:  * ### isAutoload()
279:  * 
280:  * Should this module be automatically loaded at boot?
281:  * If this method is not present, it will be auto-determined based on module type. If it is provided
282:  * in the module information array (discussed above) that will override this method.
283:  *
284:  * This method exists primarily so that base module types may specify an autoload state and have it
285:  * automatically inherit to any modules extending the type. If you are not extending a base module
286:  * type then you can implement this method, or you can provide it in your module info array. 
287:  *  
288:  * A module that returns TRUE is referred to as an "autoload" module, because it automatically loads as
289:  * part of ProcessWire's boot process. Autoload modules load before PW attempts to handle the web request.
290:  *  
291:  * Return TRUE if this module is automatically loaded at runtime.
292:  * Return FALSE if this module must be requested via `$modules->get('ModuleName')` method before it is loaded.
293:  *  
294:  * Modules that are intended to attach hooks in the application typically should be autoload because
295:  * they listen in to classes rather than have classes call upon them. If they weren't autoloaded, then
296:  * they might never get to attach their hooks.
297:  *  
298:  * Modules that shouldn't be autoload are those that may or may not be needed at runtime, for example
299:  * `Fieldtype` and `Inputfield` modules.
300:  *  
301:  * _As a side note, I can't think of any reason why a non-singular module would ever be autoload. The fact that
302:  * an autoload module is automatically loaded as part of PW's boot process implies it's probably going to be the
303:  * only instance running. So if you've decided to make your module 'autoload', then is safe to assume you've
304:  * also decided your module will also be singular (if that helps anyone reading this)._

306:  * ### install()
307:  * 
308:  * This method is called when the module is first installed. If implemented, install() methods typically are
309:  * defined hookable as `public function ___install()`. 
310:  * 
311:  * The method should prepare the environment with anything else needed by the module, such as newly created 
312:  * fields, pages, templates, etc. or installation of other modules. 
313:  * 
314:  * If the install() method determines that the module called be installed for some reason, it should 
315:  * throw a `WireException.` 
316:  * 
317:  * ### uninstall()
318:  * 
319:  * This method is called when the module is uninstalled. If implemented, uninstall() methods typically are
320:  * defined hookable as `public function ___uninstall()`. 
321:  * 
322:  * This method should undo everything done by the install() method, or undo anything created by the module,
323:  * restoring the system back to the state that it was in before the module was installed. 
324:  * 
325:  * If the uninstall() method determines that it cannot proceed for some reason, it should throw 
326:  * a `WireException`. 
327:  * 
328:  * ### upgrade($fromVersion, $toVersion)
329:  * 
330:  * This method is called when a version change is detected. This method should make any adjustments needed
331:  * to support the module from one version to another. The previous known version ($fromVersion) and new
332:  * version ($toVersion) are provided as arguments.
333:  * 
334:  * If implemented, uninstall() methods typically are defined hookable as `public function ___upgrade(...)`. 
335:  * If the upgrade cannot proceed for some reason, this method should throw a `WireException`. 
336:  *

339:  *
340:  * #pw-body
341:  * 
342:  * The following methods may or may not be implemented, all are optional:
343:  * 
344:  * #pw-method void install() Called when module is installed. 
345:  * #pw-method void uninstall() Called when module is uninstalled. 
346:  * #pw-method void upgrade($fromVersion, $toVersion) Called when a version change is detected for the module. 
347:  * #pw-method array getModuleInfo() Static method that returns array of module info (not required if module implements an info file instead). 
348:  * #pw-method void init() Called when the module is loaded, immediately after any configuration data has been populated to it. 
349:  * #pw-method void ready() For autoload modules only, called when the ProcessWire API is ready to use. 
350:  * #pw-method void setConfigData(array $data) Modules may optionally provide this method to receive config data from ProcessWire.
351:  * #pw-method bool isSingular() #pw-internal
352:  * #pw-method bool isAutoload() #pw-internal
353:  *

499: /**
500:  * Interface SearchableModule
501:  *
502:  * Interface for modules that implement a method and expected array return value
503:  * for completing basic text searches (primarily for admin search engine).
504:  * 
505:  * It is optional to add this interface to "implements" section of the module class definition.
506:  * However, you must specify a "searchable: name" property in your getModuleInfo() method in 
507:  * order for ProcessWire to recognize the module is searchable. See below for more info:
508:  *
509:  * ~~~~~~
510:  * public static function getModuleInfo() {
511:  *   return array(
512:  *     'searchable' => 'name', 
513:  * 
514:  *     // You'll need the above 'searchable' property returned by your getModuleInfo(). 
515:  *     // The value of 'name' should be the name by which search results should be referred to  
516:  *     // if the user wants to limit the search to this module. For instance, if your module 
517:  *     // was called ProcessWidgets, youd probably choose the name widgets for this. 
518:  *     // If the module represents an API variable, the name should be the same as the API variable. 
519:  *     // ...
520:  *   );
521:  * }
522:  * ~~~~~
523:  * 
524:  */
525: interface SearchableModule {
526: 
527:     /**
528:      * Search for items containing $text and return an array representation of them
529:      * 
530:      * You may also implement this method as hookable, i.e. ___search(), but note that youll
531:      * want to skip the "implements SearchableModule" in your class definition. 
532:      *
533:      * Must return PHP array in the format below. For each item in the 'items' array, Only the 'title' 
534:      * and 'url' properties are required for each item (the rest are optional). 
535:      *
536:      * $result = array(
537:      *   'title' => 'Title of these items',
538:      *   'total' => 999, // total number of items found, or omit if pagination not supported or active
539:      *   'url' => '', // optional URL to view all items, or omit for a PW-generated one
540:      *   'properties' => array(), // optional list of supported search properties, only looked for if $options['info'] === true;
541:      *   'items' => array(
542:      *     [0] => array(
543:      *       'id' => 123, // Unique ID of item (optional)
544:      *       'name' => 'Name of item', // (optional)
545:      *       'title' => 'Title of item', // (*required)
546:      *       'subtitle' => 'Secondary/subtitle of item',  // (optional)
547:      *       'summary' => 'Summary or description of item', // (optional)
548:      *       'url' => 'URL to view or edit the item', // (*required)
549:      *       'icon' => 'Optional icon name to represent the item, i.e. "gear" or "fa-gear"', // (optional)
550:      *       'group' => 'Optionally group with other items having this group name, overrides $result[title]', // (optional)
551:      *       'status' => int, // refers to Page status, omit if not a Page item (optional)
552:      *       'modified' => int, // modified date of item as unix timestamp (optional)
553:      *     [1] => array(
554:      *       ...
555:      *     ),
556:      *   ),
557:      * );
558:      *
559:      * PLEASE NOTE:  
560:      * When ProcessWire calls this method, if the module is not already loaded (autoload), 
561:      * it instantiates the module but DOES NOT call the init() or ready() methods. Thats because the 
562:      * search method is generally self contained. If you need either of those methods to be called,
563:      * and your module is not autoload, you should call the method(s) from your search() method.
564:      * 
565:      * About the optional properties index:
566:      * If ProcessWire calls your search() method with $options['info'] == true; then it is likely wanting to see
567:      * what properties are available for search. For instance, properties for a Module search might be: 
568:      * [ 'name', 'title', 'summary' ]. Implementation of the properties index is optional, and for PWs informational
569:      * purposes only. 
570:      * 
571:      * @param string $text Text to search for
572:      * @param array $options Options array provided to search() calls: 
573:      *  - `edit` (bool): True if any 'url' returned should be to edit rather than view items, where access allows. (default=true)
574:      *  - `multilang` (bool): If true, search all languages rather than just current (default=true).
575:      *  - `start` (int): Start index (0-based), if pagination active (default=0).
576:      *  - `limit` (int): Limit to this many items, or 0 for no limit. (default=0).
577:      *  - `type` (string): If search should only be of a specific type, i.e. "pages", "modules", etc. then it is 
578:      *     specified here. This corresponds with the getModuleInfo()['searchable'] name or item 'group' property. 
579:      *     Note that ProcessWire wont call your search() method if the type cannot match this search. 
580:      *  - `operator` (string): Selector operator type requested, if more than one is supported (default is %=).
581:      *  - `property` (string): If search should limit to a particular property/field, it is named here. 
582:      *  - `verbose` (bool): True if output can optionally be more verbose, false if not. (default=false)
583:      *  - `debug` (bool): True if DEBUG option was specified in query. (default=false)
584:      *  - `help` (bool): True if we are just querying for help/info and are not using the search results. (default=false)
585:      * @return array
586:      *
587:      */
588:     public function search($text, array $options = array()); 
589: }
